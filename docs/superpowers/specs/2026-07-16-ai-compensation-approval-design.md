# AI 智能补偿审批设计

日期：2026-07-16

## 目标

在既有 GameOps 玩家工单和执行闭环中，为已核验的“充值成功但返利或奖励漏发”申请加入 AI 智能审批。中低风险申请自动通过，高风险、资料不全或模型异常的申请保留人工审批。

自动审批必须具有明显、可检索、不可伪装成人工审批的 AI 痕迹。系统不直接发放奖励；它只决定是否让现有的补发执行任务进入可执行状态。

## 现有系统与集成边界

本功能复用下列既有能力：

- `omnigent/gameops/ticket_triage_agent.py`：将支付和返利问题识别为 `payment_reward` 工单，并生成核验和例外审批任务。
- `omnigent/gameops/execution.py`：持久化任务状态、阻止未审批任务执行、记录审批和执行审计。
- `omnigent/server/routes/gameops.py`：提供 GameOps API 路由和 API Key 保护。
- `omnigent/gameops/llm_client.py`：作为第一方业务运行时的模型边界。

本功能不新建独立的审批数据库或绕开 `GameOpsExecutionRuntime`。现有 `POST /gameops/execution/approve` 继续只处理人工审批。

## 适用范围

第一版仅适用于满足以下条件的单玩家漏发返利/奖励补发申请：

- 工单类别为 `payment_reward`。
- 已提供玩家 ID、服务器 ID、订单 ID、活动 ID 和发生时间。
- 已提供订单支付成功、活动资格通过、奖励发放失败三项核验结果。
- 申请是补发已承诺的活动奖励，而不是退款、善意补偿、账号操作、封禁申诉或全服补偿。

退款、账号操作、活动配置错误、服务事故补偿、跨服例外和全服补偿均不属于自动审批范围，直接转人工。

## 决策策略

### 硬规则

以下任一条件命中时，系统不调用 AI 自动通过，直接将任务保持为 `waiting_approval`：

- 必填字段或三项核验证据缺失、冲突或不可验证。
- 订单未支付、活动资格不通过，或奖励日志并非明确的发放失败。
- 账号存在处罚、异常或安全复核标记。
- 申请涉及退款、现金价值补偿、跨服例外、活动配置错误或服务事故。
- 涉及高级货币且价值高于补偿政策规定的 500 gems 人工审批门槛。
- 最近 30 天已有两次或以上人工补偿记录。

这些规则高于任何模型输出。AI 不得改变资格、金额或规则阈值。

### AI 研判

硬规则通过后，AI 仅根据结构化的工单、玩家资料、核验证据和已检索到的政策片段生成严格 JSON：

```json
{
  "risk_level": "low | medium | high",
  "risk_score": 0,
  "recommended_action": "auto_approve | manual_review",
  "reason": "基于证据的简短判断",
  "evidence_used": ["订单支付成功", "活动资格通过", "发奖日志失败"]
}
```

只有 `low` 或 `medium`、`auto_approve`，且所有硬规则均通过时，系统才自动批准。输出非 JSON、字段缺失、值域错误、超时或模型不可用时一律转人工。

## API 与数据模型

新增 `POST /v1/gameops/tickets/approval/evaluate`。请求包含原始 `ExecutionTask`、玩家资料快照、补发类型与数量、订单/资格/发奖核验证据，以及可选的操作批次标识。

响应返回更新后的任务和独立的 `AiApprovalDecision`：

- `decision_id`
- `decision_source: ai_auto | manual | rule_blocked | fallback`
- `decision_status: auto_approved | manual_review`
- `risk_level` 与 `risk_score`
- `reason` 与 `evidence_used`
- `hard_rule_results`
- `model_id`、`prompt_version`、`decided_at`

`ExecutionTask` 增加审批来源和决策 ID，避免把 AI 写入 `approved_by` 并伪装成真实审批人。自动通过的任务转为 `pending`，人工任务保持 `waiting_approval`。执行层只信任已保存的审批来源与决策状态，不信任客户端提交的“已通过”字段。

审计历史新增 `ai_approve` 事件类型。它与 `approve`（人工审批）和 `run`（执行）分开保存，并包含完整决策摘要和证据快照。

## 明显的 AI 审批痕迹

工单页面和执行历史必须明显区分三种结果：

- `AI 自动通过`：显示模型、风险等级/分数、理由、使用证据和决策时间。
- `转人工审批`：显示触发的硬规则或 AI 高风险理由，并保留人工审批入口。
- `人工审批结果`：显示真实审批人、人工备注和操作时间。

AI 自动通过任务在状态标签、任务详情和审计记录中均显示 `AI 自动审批`，不得使用人名、`approved_by` 或“人工已审批”等文案。详情页提供决策 ID，供运营按时间、玩家、任务或决策来源检索。

## 处理流程

1. 调用既有工单分诊接口，生成 `payment-log-check` 和 `exception-approval` 任务。
2. 操作员提交订单状态、活动资格、奖励日志和玩家资料等核验材料。
3. 新审批接口运行硬规则；命中时写入 `rule_blocked` 审计并转人工。
4. 未命中时，AI 审核器基于证据和政策输出结构化建议。
5. 合并规则与 AI 输出：符合条件则写入 `ai_auto` 决策、将审批任务置为可执行；否则写入 `manual_review`。
6. 现有执行接口在执行前继续检查证据、审批状态和操作角色，并记录补发工具的回执。
7. 人工可通过既有人工审批接口处理转人工的任务；人工结果追加审计，不覆盖 AI 或规则记录。

## 安全与失败处理

- AI 只给出补发流程的批准建议，不执行外部游戏后台操作。
- 模型提示词只传递最小必要资料；不传递 API 密钥或无关隐私数据。
- 所有故障采用 fail closed：转人工而不是自动通过。
- AI 的理由必须可回溯至传入证据和已检索政策，不允许产生未提供的事实。
- 规则、模型和人工决策均保存不可变审计事件。

## 测试与验收

- 核验完整且低/中风险的漏发返利申请自动通过，任务显示 `AI 自动审批`。
- 响应和审计中均能查询模型、风险分、理由、证据、提示词版本和决策 ID。
- 每条硬规则均将任务转人工，且不会调用 AI 自动通过。
- AI 高风险、超时、格式错误或模型不可用时均安全转人工。
- 人工审批保留真实审批人与备注，且不覆盖先前 AI 或规则决策。
- 未通过 AI 自动审批或人工审批的任务无法被执行接口执行。
- API Key 已配置时，新接口受到与现有 GameOps 路由相同的保护。

## 非目标

- 实际接入游戏后台发奖、支付系统或 CRM。
- 用 AI 直接发放奖励、退款或修改账号状态。
- 处理批量补偿、全服补偿或账号安全类审批。
- 训练或微调模型。
